Pod
コンテナを動かす最小単位。1つ以上のコンテナをまとめたグループ。固有のIPを持つが再起動で変わる。
Deployment
Podを管理する仕組み。「何台動かすか」「どのイメージか」を宣言的に指定。自己修復する。
Service
Podへのネットワーク窓口。Podが再起動してもIPが変わらない固定エンドポイントを提供。ロードバランシングも担う。
03 / 06

resource #1

Pod — コンテナの最小単位

Pod: mynginx
コンテナ (nginx)
実際のアプリが動く場所
IPアドレス: 10.x.x.x ← 再起動で変わる
💡 なぜコンテナを直接使わないか
PodはIPアドレスや起動設定をコンテナにまとめて与える。複数コンテナを同じネットワーク・ストレージで動かしたい場合もPodでグループ化する。
起動 / 確認 / 削除
# Pod を起動
kubectl run mynginx --image=nginx
# 一覧確認
kubectl get pods
NAME READY STATUS AGE
mynginx 1/1 Running 10s
# 削除
kubectl delete pod mynginx
詳細 / ログ / exec
kubectl describe pod mynginx
kubectl logs mynginx
kubectl exec -it mynginx -- bash
⚠️ 注意: Podを直接作ると削除したら終わり。
再起動・複数台管理には Deployment を使う。

🔄 ローリングアップデート
古いPodを1台ずつ新しいイメージに置き換える。
ダウンタイムなしで更新できる。
失敗したら kubectl rollout undo で即ロールバック。
作成 / 確認
kubectl create deployment web --image=nginx --replicas=3
kubectl get deployments
kubectl get pods
スケール / 更新 / ロールバック
# 5台に増やす
kubectl scale deployment/web --replicas=5
# イメージを更新
kubectl set image deployment/web nginx=nginx:alpine
kubectl rollout status deployment/web
# 元に戻す
kubectl rollout undo deployment/web
05 / 06

resource #3

Service — Podへの固定窓口

クライアント
↓ 固定IP・ポートで接続
🔌 Service（固定エンドポイント）
↓ ロードバランシング
Pod 1  /  Pod 2  /  Pod 3
ClusterIP
クラスタ内部のみ（デフォルト）
NodePort
ノードのポートで外部公開
LoadBalancer
クラウドのLBを使う（本番向け）
Deployment に Service を作る
kubectl create deployment web --image=nginx --replicas=2
kubectl expose deployment web --type=NodePort --port=80
kubectl get services
NAME TYPE CLUSTER-IP PORT(S)
web NodePort 10.96.x.x 80:3xxxx/TCP
⚡ kind でブラウザアクセスする方法
kindはDockerコンテナがノードなのでNodePortに直接アクセスできない。port-forward を使う。

kubectl port-forward service/web 8080:80
→ http://localhost:8080 でアクセス可能
