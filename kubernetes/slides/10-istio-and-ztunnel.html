03 / 06

features

Istio が具体的にやること

🔐
① mTLS（相互 TLS 認証）
Pod 間通信で双方向の証明書検証。
「本当に Pod B か？」をなりすまし防止。
証明書は istiod が自動発行・ローテーション（SPIFFE/SVID）。
🔒
② 暗号化
Pod 間・Node 間の通信をすべて暗号化。
アプリ側のコード変更ゼロで平文パケットをなくせる。
iptables redirect でアプリは意識しない。
🛡️
③ L4 ポリシー強制
「Service A は Service B としか通信できない」ルールを強制。
NetworkPolicy より細かい Workload Identity（SPIFFE ID） ベースで制御。
📊
④ テレメトリ収集
リクエスト数・レイテンシ・エラー率をPod に手を加えずに収集。
Prometheus / Grafana / Loki などに流せる。
ゴールデンシグナル（LETS）を自動取得。

ambient mesh

ZTunnel — Ambient Mesh モード

サイドカー vs ZTunnel
サイドカー (Envoy)ZTunnel (Ambient)
配置Pod 内に注入Node 単位 (DaemonSet)
担当レイヤーL4 + L7L4 のみ
Pod への影響あり（再起動必要）なし
L7 機能ありWaypoint に委譲
リソース効率Pod数分だけ EnvoyNode 1つに 1プロセス
実装言語C++Rust（軽量）

ZTunnel の通信フロー

Pod A
↓iptables redirect（アプリは意識しない）
ZTunnel (Node A)
↓HBONEトンネル（mTLS 暗号化済み）
ZTunnel (Node B)
↓復号・Workload ID 検証
Pod B
HBONE（HTTP-Based Overlay Network Encapsulation）:
ZTunnel 間をつなぐ独自トンネルプロトコル。mTLS を内包する。

いつ Waypoint を使うか

ユースケースZTunnel のみWaypoint 追加
mTLS・暗号化✓不要
L4 認証・認可✓不要
HTTP パスルーティング✗必要
リトライ・サーキットブレーカー✗必要
ヘッダー操作✗必要
⚠️ SRE 観点の注意
Waypoint は普通の Pod。障害時は対象 Namespace の L7 ポリシーが効かなくなる（L4 は ZTunnel が継続）。
istioctl proxy-status で設定同期状態を確認。
06 / 06

control plane & sre

コントロールプレーン istiod + SRE 活用

🧠 istiod の役割
▸証明書の発行・配布（CA: Certificate Authority）
▸ポリシーの配布（AuthorizationPolicy など）
▸ZTunnel / Waypoint Proxy への設定プッシュ（xDS プロトコル）
診断コマンド
# ZTunnel の状態確認
istioctl ztunnel-config workload
 
# Waypoint の確認
kubectl get gateway -n <namespace>
 
# 設定同期状態
istioctl proxy-status

SRE 観点での活用ポイント

🔍 障害切り分け
Pod 間通信失敗 → istioctl ztunnel-config でどのノードのどの ZTunnel が問題か特定できる。
📈 可観測性
アプリ改修なしにゴールデンシグナル（Latency / Traffic / Errors / Saturation）取得。Prometheus・Grafana へ自動連携。
🕵️ セキュリティ監査
mTLS 強制により「誰が誰と通信したか」をログで追える。インシデント時の通信経路の特定に有効。
まとめ: Istio/ZTunnel は Kubernetes の上に乗るセキュリティ・可観測性レイヤー。L7 が不要なら ZTunnel のみで十分、必要なら Waypoint Proxy を追加する。
