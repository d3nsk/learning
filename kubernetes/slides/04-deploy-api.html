03 / 06

step ① — code

サンプル API を作る

api/main.go (Go)
package main
 
import (
  "encoding/json"
  "net/http"
  "os"
)
 
func main() {
  http.HandleFunc("/healthz", func(...) {
    w.WriteHeader(http.StatusOK)
    w.Write([]byte("ok"))
  })
 
  http.HandleFunc("/hello", func(...) {
    hostname, _ := os.Hostname()
    // hostname = Pod 名が入る
    json.NewEncoder(w).Encode(map[string]string{
      "message": "Hello from Kubernetes!",
      "hostname": hostname,
    })
  })
  http.ListenAndServe(":8080", nil)
}
api/Dockerfile
# ① ビルドステージ
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY main.go .
RUN go mod init myapi && go build -o myapi .
 
# ② 実行ステージ（軽量）
FROM alpine:3.19
WORKDIR /app
COPY --from=builder /app/myapi .
EXPOSE 8080
CMD ["./myapi"]
💡 マルチステージビルド
golang の巨大なビルドイメージを最終イメージに含めない。alpine ベースで軽量な実行イメージを作る。

step ② ③ — build & load

イメージをビルドして kind にロード

Terminal
# ② イメージをビルド
docker build -t myapi:v1 .

# 確認

# ③ kind にロード

# ノード内で確認
docker exec -it mycluster-control-plane crictl images | grep myapi

↓ kind load docker-image

05 / 06

step ④ — deploy

Deployment + Service YAML

api-deployment.yaml
spec:
  replicas: 3
  selector:
    matchLabels: {app: myapi}
  template:
    spec:
      containers:
        - name: myapi
          image: myapi:v1
          imagePullPolicy: Never # ← kind 用
          ports:
            - containerPort: 8080
          readinessProbe: # ← Pod が Ready になる条件
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 3
            periodSeconds: 5
imagePullPolicy: Never — kind にロード済みのローカルイメージを使う。IfNotPresent だとレジストリを探す場合あり。
readinessProbe — /healthz が 200 を返すまで Traffic を流さない。
api-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: myapi
spec:
  selector:
    app: myapi
  ports:
    - port: 80
      targetPort: 8080 # コンテナのポート
apply & confirm
kubectl apply -f api-deployment.yaml
kubectl apply -f api-service.yaml
kubectl get pods -w
NAME READY STATUS
myapi-abc-001 1/1 Running
myapi-abc-002 1/1 Running
myapi-abc-003 1/1 Running
06 / 06

step ⑤ — verify & update

動作確認 +イメージ更新

動作確認
# ヘルスチェック
curl localhost/api/healthz
ok
 
# メインエンドポイント
curl localhost/api/hello
{"hostname":"myapi-abc-001","message":"Hello from Kubernetes!"}
 
# 複数回叩く → ロードバランシング確認
for i in $(seq 5); do curl -s localhost/api/hello | grep hostname; done
"hostname": "myapi-abc-001"
"hostname": "myapi-abc-002"
"hostname": "myapi-abc-003" ← Pod が変わっている！
hostname が変わる = 複数 Pod にリクエストが分散されている証拠。
kubectl scale deployment myapi --replicas=5 で増やして再確認できる。

イメージ更新フロー（v2 リリース）

1
コードを修正して再ビルド
docker build -t myapi:v2 .
2
kind にロード
kind load docker-image myapi:v2 --name mycluster
3
Deployment を更新（ローリング）
kubectl set image deployment/myapi myapi=myapi:v2
4
進捗確認
kubectl rollout status deployment/myapi
