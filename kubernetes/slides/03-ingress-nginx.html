02 / 06

problem → solution

Service だけだと何が困るか

✗ NodePort でアプリごとにポートが増える
クライアント
├── :30080 → Service A (frontend)
├── :30081 → Service B (api)
└── :30082 → Service C (admin)
⚠️ アプリが増えるたびにポートも増える
⚠️ ポート番号を覚える必要がある
⚠️ 80/443 で公開できない
✓ Ingress でパス・ホスト名ルーティング
クライアント
└── :80
├── / → frontend
├── /api → api
└── /admin → admin
✓ ポート 80 ひとつで全アプリを公開
✓ パス / ホスト名で直感的にルーティング
✓ TLS も一括管理できる
これを実現するのが Ingress。ルーティングルールを定義する Kubernetes リソース。Ingress Controller（今回は nginx）が実際にトラフィックを処理する。

04 / 06

setup

kind クラスタ再作成 +
Ingress Controller インストール

kind-config.yaml
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
  - role: control-plane
    kubeadmConfigPatches:
      - |
        nodeRegistration:
          kubeletExtraArgs:
            node-labels: "ingress-ready=true"
    extraPortMappings:
      - containerPort: 80
        hostPort: 80 # ← ポイント
      - containerPort: 443
        hostPort: 443
1
既存クラスタを削除
kind delete cluster --name mycluster
2
設定ファイルでクラスタを再作成
kind create cluster --name mycluster --config kind-config.yaml
ポート 80/443 をホストに公開するのがポイント
3
nginx Ingress Controller をインストール
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/kind/deploy.yaml
4
起動を待つ
kubectl wait --namespace ingress-nginx --for=condition=ready pod --selector=app.kubernetes.io/component=controller --timeout=90s
05 / 06

deploy

アプリデプロイ +
Ingress リソース作成

2つのアプリをデプロイ
# frontend: nginx のデフォルトページ
kubectl create deployment frontend --image=nginx
kubectl expose deployment frontend --port=80
 
# api: httpbin で JSON を返す
kubectl create deployment api --image=kennethreitz/httpbin
kubectl expose deployment api --port=80
動作確認
# / → frontend
curl localhost/
Welcome to nginx!
# /api → api (httpbin)
curl localhost/api/get
{"url": "http://localhost/api/get"}
ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
    - http:
        paths:
          - path: /  # → frontend
            pathType: Prefix
            backend:
              service: {name: frontend, port: {number: 80}}
          - path: /api # → api
            pathType: Prefix
            backend:
              service: {name: api, port: {number: 80}}
kubectl apply -f ingress.yaml
06 / 06

verify & summary

ルーティングの仕組みまとめ

/api/get へのリクエストの流れ

クライアント curl localhost/api/get
↓
Ingress Controller /api にマッチするルールを検索
↓ /api → Service: api
Service: api ClusterIP でPodへ
↓
Pod (httpbin) レスポンスを返す
💡 rewrite-target: / アノテーション
/api/get → /get に書き換えてアプリへ届ける。
プレフィックスをアプリ側に意識させないための設定。

学んだこと

✓ Service の NodePort 方式ではポートが増え続けるという課題を理解した
✓ Ingress Controller（nginx）がルーティングを処理する実体
✓ Ingress リソースでパスとService の対応を YAML で定義する
✓ kind では extraPortMappings で80番をホストに公開する必要がある
✓ rewrite-target アノテーションでパスプレフィックスを除去できる
